SecureCorp Lab Guide

Follow the phases in order. Each phase tells you what to do, what to collect, and has expandable step-by-step hints if you get stuck.

Hidden pages: /check (submit findings) · /guide (this page)

Progress checklist — 19 items

Every labeled value you find maps to a field on /check. Collect these as you go:

Auth Username · Bypass method · Admin password · Dashboard flag · Internal token · Cookie role
Web attacks XSS canary · Stored XSS (yes) · IDOR · LFI · CMDi · SSRF · Backup flag · Reset flag · SSTI · Profile flag · Report SQLi
Host Host shell flag · Host root flag

Before you start

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
VULNLAB_RESET=1 python app.py

Open http://127.0.0.1:5000

Recommended tools: Browser DevTools (Network + Application tabs), Burp Suite or OWASP ZAP, curl.

Phase 1 — Reconnaissance

Goal: Map every page, API, and leaked file before attacking.

  1. Click through every link on the homepage and navigation bar. Write down each URL.
  2. On the sign-in page, right-click → View Page Source. Look for comments and hidden fields.
  3. Visit /robots.txt in your browser — note every disallowed path.
  4. Try /api/status and read the JSON response.
  5. Open Company Documents and follow links to the file viewer.
Collect now: Backup config flag · XSS canary flag (both in backup file)

Done when: You have read /backup/config.bak and saved both labeled values from it.

Step-by-step hints — Phase 1
1. Find robots.txt
  1. Go to http://127.0.0.1:5000/robots.txt
  2. Read the Disallow: lines — these are paths the site owner tried to hide
  3. Note /backup/, /internal/, /sys/, /api/user/, /api/reports
2. Read the leaked backup config
  1. Go to http://127.0.0.1:5000/backup/config.bak
  2. Find lines labeled Backup config flag and XSS canary flag
  3. Copy both VULN{...} values — you will enter them on /check later
  4. Also note INTERNAL_HEALTH_URL — you will need it in Phase 7
3. Browse documents
  1. Click Documents in the nav (or go to /documents)
  2. Click each filename — they open in /files?name=...
  3. Read deployment_notes.txt — it explains host access (Phase 9)
4. Guest account in page source
  1. Go to /login → View Source
  2. Search for guest — a developer comment reveals test credentials

Phase 2 — Login & admin access

Goal: Reach the administrator dashboard and copy all labeled secrets.

  1. Sign in with wrong passwords for admin vs notreal — compare error messages.
  2. Intercept the login POST — find the hidden role field.
  3. Get admin dashboard access (three methods below — pick one).
  4. Open My Dashboard as admin and copy every labeled value.
Collect now: Admin password · Dashboard flag · Internal token · Cookie role (admin) · Username · Bypass method

Done when: Dashboard shows "Administrator resources" with four labeled items.

Step-by-step hints — Phase 2
Method A — Hidden role field (easiest)
  1. Open Burp/ZAP or DevTools → Network tab
  2. Go to /login, enter any username/password, submit
  3. Find the POST request — body looks like: username=...&password=...&role=user
  4. Change role=user to role=admin
  5. Forward/resend the request → you land on dashboard as admin
  6. On /check, set Bypass method to hidden role
Method B — SQL injection
  1. On login, set username to: ' OR '1'='1' --
  2. Password can be anything
  3. Also change hidden role to admin via intercept (SQLi alone gives user role)
  4. On /check, set Bypass method to sqli
Method C — Cookie tampering
  1. Log in normally as guest / guest123
  2. DevTools → Application → Cookies → copy vulnlab_session
  3. Decode URL-safe base64 → JSON like {"username":"guest","role":"user"}
  4. Change to {"username":"guest","role":"admin"}, re-encode, replace cookie
  5. Reload /dashboard
  6. On /check, set Bypass method to cookie tamper and Cookie role to admin
python3 -c "import base64,json; p={'username':'guest','role':'admin'}; print(base64.urlsafe_b64encode(json.dumps(p,separators=(',',':')).encode()).decode().rstrip('='))"
User enumeration
  1. Try login with username admin + wrong password → "Invalid password for that account."
  2. Try username fakeuser → "Unknown username."
  3. Or: curl "http://127.0.0.1:5000/api/check_user?username=admin"

Phase 3 — XSS (cross-site scripting)

Goal: Prove reflected and stored XSS. XSS canary flag is from Phase 1 backup file.

  1. Directory search — submit HTML/script in the search box, see if it executes.
  2. Feedback — post a script tag, reload page, confirm it runs.
  3. Mark Stored XSS as Yes on /check.
Collect now: XSS canary flag (backup config) · Stored XSS = yes
Step-by-step hints — Phase 3
Reflected XSS
  1. Go to Directory (/search)
  2. Search for: <script>alert(1)</script>
  3. If an alert pops up, reflected XSS works
Stored XSS
  1. Go to Feedback
  2. In the message field enter: <script>alert('stored')</script>
  3. Submit → reload page → alert should fire again
  4. On /check, set Stored XSS to Yes

Phase 4 — SSTI (server-side template injection)

Goal: Inject template syntax in the directory search to leak a server-side flag.

  1. Confirm template evaluation with {{7*7}} — should show 49.
  2. Leak the flag variable with {{ssti_flag}}.
Collect now: SSTI flag
Step-by-step hints — Phase 4
Full walkthrough
  1. Go to /search?q={{7*7}}
  2. You should see "Template preview: 49" — the server evaluated your input
  3. Now try /search?q={{ssti_flag}}
  4. Copy the VULN{...} value labeled SSTI flag

Phase 5 — IDOR

Goal: Read other users' profiles by changing the ID in the URL.

  1. Visit /api/user/1, then /2, then /3.
  2. Find the admin profile and copy the labeled api_key.
Collect now: IDOR flag
Step-by-step hints — Phase 5
Full walkthrough
  1. Open http://127.0.0.1:5000/api/user/1 in browser or curl
  2. Repeat for IDs 2 and 3
  3. User 3 (admin) has api_key containing the IDOR flag
  4. No login required — classic insecure direct object reference

Phase 6 — Report API SQL injection

Goal: SQL-inject the department filter on the hidden reports API.

  1. Find /api/reports (listed in robots.txt).
  2. Try normal dept values, then injection to return all rows.
  3. When 3+ users returned, the response includes the report flag.
Collect now: Report SQLi flag
Step-by-step hints — Phase 6
Full walkthrough
  1. Try: /api/reports?dept=SOC — returns one user
  2. Inject: /api/reports?dept=' OR '1'='1' --
  3. URL-encode if using browser: %27+OR+%271%27%3D%271%27+--
  4. Response includes report_flag when all users are returned

Phase 7 — Local file read

Goal: Read protected files via the document viewer.

  1. Go to /files?name=backup_flag.txt (linked from Documents page).
  2. Copy the labeled LFI flag from the response.
Collect now: LFI flag
Step-by-step hints — Phase 7
Full walkthrough
  1. curl "http://127.0.0.1:5000/files?name=backup_flag.txt"
  2. Also read deployment_notes.txt — needed for Phase 11

Phase 8 — Command injection

Goal: Run extra commands through the network ping tool.

  1. Go to Network Tools (/tools/ping).
  2. Enter a hostname with a shell metacharacter appended.
  3. Read the labeled flag in the debug output.
Collect now: Command injection flag
Step-by-step hints — Phase 8
Full walkthrough
  1. Host field: 127.0.0.1; id
  2. Or use curl: curl -X POST http://127.0.0.1:5000/tools/ping -d "host=127.0.0.1;id"
  3. Output includes the command injection flag after [debug]

Phase 9 — SSRF

Goal: Make the server fetch an internal URL you cannot reach directly.

  1. Get INTERNAL_HEALTH_URL from the backup config (Phase 1).
  2. Open URL Preview tool (link on Network Tools page).
  3. Paste the internal URL — read ssrf_flag from the JSON response.
Collect now: SSRF flag · host console token (same response)
Step-by-step hints — Phase 9
Full walkthrough
  1. From backup config copy: http://127.0.0.1:5000/internal/health
  2. Go to URL Preview (/fetch) or: /fetch?url=http://127.0.0.1:5000/internal/health
  3. The server fetches it — you see JSON with ssrf_flag and host_console_token
  4. Save both — you need the token for Phase 11

Phase 10 — Password reset

Goal: Forge a valid reset token without email access.

  1. Generate a reset link for username admin.
  2. Notice the token pattern — it's predictable.
  3. Visit the reset URL and copy the labeled flag.
Collect now: Password reset flag
Step-by-step hints — Phase 10
Full walkthrough
  1. Go to Reset Password, enter username admin, submit
  2. Token = URL-safe base64 of username: YWRtaW4 for admin
  3. Visit: /reset?user=admin&token=YWRtaW4
  4. Copy the password reset flag from the success message

Phase 11 — Profile escalation

Goal: Tamper with a hidden field on the profile form to escalate access.

  1. Log in (any account).
  2. Go to My Dashboard then edit profile (or visit /profile while logged in).
  3. Intercept the save request and change the hidden access_level field.
Collect now: Profile escalation flag
Step-by-step hints — Phase 11
Full walkthrough
  1. Sign in as guest / guest123
  2. Go to /profile
  3. Intercept POST — body includes hidden access_level=standard
  4. Change to access_level=admin, forward request
  5. Page shows the profile escalation flag

Phase 12 — Host compromise (shell + root)

Goal: Get a web shell on the server and read both host flags.

  1. Use host_console_token from Phase 9.
  2. Open /sys/console?token=YOUR_TOKEN.
  3. Run help, then cat home/www-data/user_flag.txt.
  4. Run sudo cat root/flag.txt for root flag.
Collect now: Host shell flag · Host root flag
Step-by-step hints — Phase 12
Full walkthrough
  1. http://127.0.0.1:5000/sys/console?token=c0ns0l3_t0k_8e1a4f (replace with your token)
  2. Type ls to browse the simulated filesystem
  3. cat home/www-data/user_flag.txt → host shell flag
  4. cat home/www-data/notes.txt → explains sudo privesc
  5. sudo cat root/flag.txt → host root flag

Phase 13 — Submit everything

Goal: Score 19/19 on the hidden submission form.

  1. Go to /check (not linked anywhere on the site).
  2. Enter every labeled value under its matching field name.
  3. Stored XSS → Yes. Cookie role → admin.
  4. Submit and verify 19/19 complete.
Step-by-step hints — Phase 13
Submission form
  1. Open /check
  2. Field labels match exactly what you saw on the site (e.g. "Dashboard flag", "SSTI flag")
  3. Bypass method accepts: sqli, hidden role, cookie tamper, etc.
